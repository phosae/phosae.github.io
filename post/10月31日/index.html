更新流程

应用 Spring 容器中的什么对象会更新

  • 所有注解了 @ConfigurationProperties 的 Bean 都会自动刷新，直接或间接引用了 @ConfigurationProperties Bean 的对象会受更新影响，发生在图中步骤 5
  • 所有注解了 @RefreshScope 的 Bean 都会刷新，发生在图中步骤 6,8
  • 应用 Environment 对象存放着应用配置，所以也会更新，发生在图中步骤 3

反过来可以回答什么对象不会更新： * 没有声明 @RefreshScope, 只通过 @Value 注入配置的 Spring Bean 不会更新。 * 没有引用配置的 Spring Bean 不会更新（没有使用 @Value 注入或者没有引用 @ConfigurationProperties Bean)

因此，即使你通过配置中心更改配置，更新也仅仅止于 Environment 对象，真正负责业务的 Bean 可能不会更新

更新过程中可以监听的事件

  • RefreshEvent, 配置更新事件, 发生在 Environment 变化之前，客户端会利用该事件更新 Environment，通常情况下应用代码不应该监听此事件
  • EnvironmentChangedEvent, 环境更新完成事件，发生在 Environment 对象变化完成之后，事件中携带了当前更新过程中的配置 key 集合，可以监听此事件确认对象中涉及的 key 是否发生变化
  • RefreshScopeRefreshedEvent, @RefreshScope Bean 更新完成事件

为什么要加 @RefreshScope 或 @ConfigurationProperties 配置才能刷新？

如果不加区别更新整个 Spring 容器，这种刷新会涉及所有系统资源的释放和再绑定，约等于应用重启，耗时可能长达数分钟，动态效率会下降到与应用重启类似，倒不如回退到重启变更

声明式规范可以极大减少动态更新范围，如 @ConfigurationProperties 的 rebind 操作只涉及内存读写，更新时长为 ms 量级甚至可能是 ns 量级，@RefreshScope Bean 可能会涉及资源的释放和再绑定，更新时长量级为 s

动态更新策略参考

根据此前描述，动态更新比较适用于轻量级场景，如开关、简单阈值调整、订阅 topic 调整。

如果你打算做类似刷新具有存量任务的线程池等的行为更新，需要考虑生效延迟，并谨慎编写资源动态调整代码。

        // 业务分支 1
    }else {
        // 业务分支 2
    }

对象资源使用行为调整

适用范围：Eureka 注册中心地址修改, RocketMQ nameServer 地址端口修改, RocketMQ 订阅 topic 修改，线程池大小调整等涉及系统资源释放/重新获取的资源适用行为调整。

在此种情况下，仅仅更新内存中的某个值并不会改变程序行为，应用代码必须释放之前占用的系统资源，如释放 Socket、关闭线程池，再根据新配置重新申请 binding Socket、重新启动线程池。

此外，动态地从远程消息增加订阅或者减少订阅 topic，还涉及远程服务调用（Http, rpc)，变化结果应以远程响应为准。

因此，应用需要在 @RefreshScope 或 @ConfigurationProperties 的基础上额外编写代码实现动态性，下面列述一些可供参考的方法：

  • 事件驱动实现（推荐）：监听 EnvironmentChanged 事件(包含了变更的配置 key)和 RefreshedScopeRefreshedEvent 事件，在EnvironmentChanged 回调函数中释放资源，再在 RefreshedScopeRefreshedEvent 回调函数中重新申请资源。

    @Value("${app.subscribe.topics}")
private String topics;

@EventListener
public void onChange(EnvironmentChangeEvent e){
if(e.getKeys().contains("app.subscribe.topics")){
		String newTopics = applicationContext.getEnvironment()
        .getProperty("app.subscribe.topics", String.class,"");
    if(!newTopics.equals(topics)){
       shutdown();
    }
}
}

@EventListener
public void onRefresh(RefreshScopeRefreshedEvent e){
if(isShutdown){
    start();
    reSubscribe(topics);
}
}

  • Spring Bean 生命周期实现：每次配置刷新时 @RefreshScope Bean 都会销毁并重新生成的特性。在销毁声明周期函数中中写注销逻辑（如 PreDestroy），在构造函数中写注册逻辑，例如 com.netflix.discovery.DiscoveryClient 就采用了此种实现方式。

注意事项及总结

行为更新生命周期实现仅适用于声明为 @RefreshScope 的 Bean，无论更新事件是否涉及当前 Bean，都会随更新事件发生销毁生命周期函数调用(destroy-method等)并使用 new 创建 bean，如果涉及线程池的关闭和开启，会产生秒级损耗的损耗
